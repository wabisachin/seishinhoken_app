# プロジェクト固有ルール（Claude Code向け）

## GEPAプロンプト最適化パイプライン（`scripts/tuning/`）専用ルール

このセクションは`scripts/tuning/`配下のGEPAによる問題生成プロンプト自動最適化パイプラインにのみ適用される。
このリポジトリの他の作業（通常の機能開発・バグ修正等）には適用されない。**それ以外の作業では、
これまで通りコミットは必ずユーザーの明示的な指示があってから行うこと。**

1. **シードプロンプトは絶対に上書き・削除しない**。現行の`lib/generation.ts`の各出題形式
   （case-desc/case-term/nocase-desc/nocase-term）向け指示文は、GEPA実行前に
   `scripts/tuning/seed_prompts/`へそのままの内容でコピーして保存する。GEPAが生成する
   プロンプト候補は必ず別ファイル（世代・イテレーション番号付き）として保存し、シード
   ファイル自体を書き換えることは無い。
2. **GEPAの最適化イテレーションを1回実行するたびに、結果をコミットする**（ユーザーへの
   確認は不要、この専用パイプラインに限り自動でコミットしてよい）。理由: レポートに
   「どのcommitの結果か」を明記する必要があり、そのためには変更のたびにコミットして
   commit hashを確定させる必要があるため。
3. **レポートは毎回同じひな型（テンプレート）に従う**（詳細な章立ては
   `scripts/tuning/REPORT_TEMPLATE.md`を参照）。定量データ部分は決定的なコードで生成し、
   考察・次の方針などの定性的な部分は`gpt-5.6-sol`に書かせる。
4. レポートファイル名・本文には、実行日時・使用した生成モデル・検証時のgit commit hash・
   評価条件（サンプル数、対象カテゴリ等）を必ず明記し、後から同じ条件を再現できるようにする。
5. **フォルダ構成は固定する**（`scripts/tuning/`配下）:
   - `seed_prompts/{category}.txt` — 上書きしないシード本体
   - `prompts/{category}/gen{世代番号}.txt` — GEPAのイテレーションごとに生まれたプロンプト
     候補を、人間がそのまま読めるプレーンテキストで保存する（1候補=1ファイル）
   - `runs/{category}/{YYYYMMDD-HHMMSS}_{commit短縮hash}_{生成モデル名}.json` — その回の
     検証結果データ（定量値）
   - `runs/{category}/{YYYYMMDD-HHMMSS}_{commit短縮hash}_{生成モデル名}.md` — 対応するレポート
     （データと同じ`runs/{category}/`フォルダに、同じファイル名プレフィックスで格納し、
     どのデータとどのレポートが対応するか一目で分かるようにする）
