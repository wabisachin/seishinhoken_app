# GEPA最適化レポート ひな型

`scripts/tuning/runs/{category}/{YYYYMMDD-HHMMSS}_{commit短縮hash}_{生成モデル名}.md`
というファイル名で保存する（同名プレフィックスの`.json`が同じフォルダにある定量結果データ）。
毎回このひな型の章立てに従うこと（章の追加・省略はしない）。1〜3は決定的なコード
（`benchmark_report.ts`相当）が生成する。4〜5は`gpt-5.6-sol`が、1〜3のデータを入力として書く。
その回のGEPAイテレーションで生まれたプロンプト候補本体は`scripts/tuning/prompts/{category}/gen{N}.txt`
に人間が読めるプレーンテキストで保存する（レポートの6章から参照する）。

---

## 1. 実行条件

- 実行日時（ISO 8601）
- 対象カテゴリ（case-desc/case-term/nocase-desc/nocase-term のいずれか）
- 生成モデル（例: gpt-5.6-luna / gemini-3.6-flash）／検査モデル（gpt-5.6-luna固定）
- 判定役モデル（識別テスト用、gpt-5.6-luna）／代理受験者モデル（難易度ベンチマーク用、gpt-5.4-nano等）
- git commit hash（この結果が対応する時点のコード）
- GEPAイテレーション番号、Pareto候補数
- 評価サンプル数（過去問側・生成側それぞれ、識別テストの参考例数も含む）

## 2. 定量結果サマリ（表形式）

Paretoフロントの5軸それぞれのスコア:
1. マクロ正答率の一致度
2. 誤答理由分布の一致度
3. 「惜しい」選択肢数分布の一致度
4. 正答確信度分布の一致度
5. 識別テストの見破られにくさ（本物→本物判定率 と 偽物→本物判定率の差）

## 3. 前回イテレーションからの差分

各軸のスコアの増減（表形式）。初回イテレーションの場合は「初回のため差分なし」と明記。

## 4. 考察（gpt-5.6-solが記述）

何が改善し何が悪化したか、その理由として考えられること。識別テストで偽物と判定された
理由・本物なのに誤って偽物と判定された理由から読み取れる、生成プロンプトの具体的な弱点。

## 5. 次のイテレーションで検討すべき方針（gpt-5.6-solが記述）

次に試すべきプロンプト修正の方向性、注意すべきトレードオフ（Pareto上のどの軸を優先すべきか等）。

## 6. 生データへの参照

詳細ログ・個々の判定結果・使用したプロンプト本文へのファイルパス一覧。
